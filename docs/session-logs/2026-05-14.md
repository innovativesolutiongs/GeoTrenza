## Key decisions

- **Stage 4 timeline:** full schema rewrite happens in Stage 4 with proper transition logic for maintained alarm bits. Phase B does not attempt cross-packet state tracking — those bits log only.
- **Phase B handler shape:** each 0x0200 packet produces one row in `positions`, zero-or-more rows in `events`, plus an UPDATE to `devices.last_seen_at`. All three in a single transaction; rollback on any failure.
- **Event emission policy in Phase B:** one-shot alarm bits (SOS, power off, ignition on/off, vibration, tow, harsh braking, etc.) become event rows. Maintained alarm bits (overspeed, fatigue, GNSS faults, etc.) log only — events come in Stage 4 once per-device state cache exists.
- **Telemetry JSONB keys:** camelCase from `extraMessageParser` output, no transformation. `telemetry.gsmSignal`, `telemetry.batteryPercent`, `telemetry.imei`, `telemetry.wakeUpSource`, etc.
- **`device_id` resolution:** at TCP connection auth time (0x0102), not per-packet. Stored on connection-state object for lifetime of the connection.
- **`last_seen_at` update:** per-packet UPDATE in Phase B. Batched async updates deferred to Stage 4.
- **Scale revisions to Stage 1 schema:**
  - Telemetry cap tightened from 8 KB → **2 KB** (at 100M rows scale, 8 KB average = 800 GB of telemetry storage; 2 KB still comfortably fits realistic Mobicom emissions).
  - GIN index on `positions.telemetry` **required from day 1**, not deferred. Cost to build on empty table is zero; cost to build later on 10M rows is hours of downtime.
  - In-memory LRU cache for device auth — deferred to Stage 4.
  - Batched async `last_seen_at` updates — deferred to Stage 4.
- **Entity file conventions (locked):**
  - EntitySchema (object-based JS), not decorators
  - PascalCase entity `name`, snake_case `tableName`
  - **snake_case property names** for v2 entities (matches schema-v2.md and the newer in-codebase convention)
  - PKs use `bigint` with `generated: "increment"` (matches `bigserial`)
  - No defaults, FKs, indexes, or constraints in entity files — migrations are the single source of truth
  - Tight whitespace (no blank lines between columns)
- **Bigint-as-string in handler code:** all TypeORM bigint columns are treated as JS strings throughout handler code. No `parseInt`. Saved as a feedback memory for the project.
- **Legacy entity registration:** kept in `ormconfig.js` `entities` array through step 4. Strip happens in step 4 alongside handler rewrite, not in step 2 (single-responsibility commits).
- **No side quests until testing phase:** flag adjacent concerns (e.g., `DB_PASSWORD || "12345"` fallback on `ormconfig.js:21`) but don't fix them mid-implementation. Saved as a feedback memory.

### What got done (continuation)

- **Sub-step 1 — `parseLocationPacket()` utility** (`ccec15c`) — canonical decode contract: takes raw packet body bytes, returns a structured object with `recordedAt`, `latitude`, `longitude`, `speed`, `direction`, `altitude`, `alarmBits`, `statusBits`, and `extras` (merged TLV map). First sub-step of Step 4.
- **Sub-step 2 — `buildPositionRow()` utility** (`4b81f70`) — maps a parsed packet to a v2 `positions` row payload. Typed columns get explicit assignments; everything else in `extras` (minus the typed-column TLVs) sweeps into `telemetry` JSONB. Raw `alarmBits` and `statusBits` always present in `telemetry`. Mileage km → m via `*1000`. 8 unit tests covering happy path, empty extras, conversion, bigint passthrough, distinct Date refs, unknown-TLV flow, falsy-inclusion, and the sweep-all-non-typed contract.
- **Sub-step 3 — `detectOneShotEvents()` utility** (`7415947`) — declarative `ONE_SHOT_ALARM_KINDS` table (16 bits mapped to event kinds) + `MAINTAINED_ALARM_BITS` set (9 bits). Iterates 0–31, uses `(alarmBits >>> bit) & 1` for sign-safe testing. Returns `{ events, maintainedBitsSet }`; one-shots emit event payloads with `started_at === ended_at === recordedAt`. Reserved bits (3, 13, 22–26) ignored. 10 unit tests including bit-31 sign safety and the bit-13 reserved case.
- **Sub-step 4 — `executeTransaction()` utility** (`f560887`) — caller-provided `queryRunner`; wraps `startTransaction → insert(Position) → [insert(Event) if any] → update(Device) → commitTransaction` with a try/catch that only rollbacks-then-rethrows. Returns `void`. First I/O code in Stage 2. 8 unit tests using a `jest.fn()`-mocked queryRunner with a `calls` array for ordering assertions — covers happy paths (no events / with events / empty array), each failure point (position/events/device), bigint position_id passthrough, and `last_seen_at` propagation.
- **Sub-step 5 — `resolveDeviceId()` + `UnknownDeviceError`** (`8666fed`) — `findOne({ where: { terminal_id } })` returning `device.id`, throws `UnknownDeviceError(terminalId)` on null. New directory `backend/ingestion/errors/` for error classes. 6 unit tests with mocked repo.

Test count progression: 25 → 33 → 43 → 51 → 57.

### Key decisions (continuation)

- **Pure-function decomposition validated end-to-end.** All four pure utilities (`parseLocationPacket`, `buildPositionRow`, `detectOneShotEvents`, `resolveDeviceId`) tested cleanly with synthetic fixtures, no real DB/parser dependencies. Confirms the function-decomposition approach from the Step 4 design doc.
- **Sign-safe bit testing pattern locked.** `(alarmBits >>> bit) & 1` over `alarmBits & (1 << bit)`. JS bitwise treats values as signed int32; `>>>` is the unsigned form. Bit 31 (harsh turn) would silently sign-extend otherwise. Documented in `detectOneShotEvents.js` and verified by Test 5 (`0x80000000`).
- **Sweep-all-non-typed telemetry contract.** `buildPositionRow` uses an explicit `EXCLUDED_FROM_TELEMETRY` set (the four typed-extra keys), then loops `Object.entries(extras)` and writes anything not in that set straight into `telemetry`. New TLVs added to `extraMessageParser` auto-flow into `telemetry` JSONB without code change here. Test 8 of `buildPositionRow.test.js` locks the contract by asserting three non-listed parser keys (`fuelAnalog`, `rotation`, `loadWeight`) all land in telemetry.
- **Telemetry keeps raw `alarmBits` and `statusBits` integers.** Cheap (8 bytes total), future-proof for Stage 4 transition-tracking back-population, and gives debug-grade visibility on every position row.
- **Conditional typed-column assignment (no `undefined` values).** Typed-extra columns (`satellites`, `signal_strength`, `battery_voltage`, `mileage_m`) are only set on the row when the TLV is present. Test 2 of `buildPositionRow.test.js` asserts via `Object.keys` that absent TLVs produce *absent keys*, not `undefined`-valued keys — important for the downstream INSERT semantics.
- **`executeTransaction` design lock-in (decisions 4A/4D/4F):** caller owns connect/release of the queryRunner; function returns `void` (no `position_id` exposed yet — add when a caller needs it); errors propagate naturally, try/catch only exists to call `rollbackTransaction` before rethrow. Error categorization (parse vs db vs unknown-device) is the orchestrator's job per the design doc.
- **Transaction order of operations:** position INSERT → events INSERT (batched, with captured `position_id`) → device UPDATE → commit. Captured `position_id` flows from `result.identifiers[0].id` and gets spread onto every event payload before the batched insert.
- **bigint-as-string preservation tested at every boundary.** 2^53 + 1 (`"9007199254740993"`) used as the canonical test value across `buildPositionRow`, `detectOneShotEvents`, `executeTransaction`, and `resolveDeviceId`. Each function guards against accidental `parseInt`.
- **`UnknownDeviceError` created early.** Sub-step 6 was scheduled to land the full error-class set + `ERROR_CATEGORIES` table; pulled `UnknownDeviceError` forward to sub-step 5 since it's tightly coupled to `resolveDeviceId` and made the test contract cleaner. Sub-step 6 now smaller in scope.
- **Mock queryRunner pattern established.** `jest.fn()` spies on `startTransaction/commitTransaction/rollbackTransaction/manager.insert/manager.update` plus a `calls` array for ordering assertions. Reusable for any future I/O-path testing in this stage.
- **Test fixtures are synthetic, not parser-derived.** All sub-step 2–5 tests use hand-built `packetData` objects, not hex bytes piped through `parseLocationPacket`. Decouples utility tests from parser changes — the parser has its own test suite (sub-step 1).
