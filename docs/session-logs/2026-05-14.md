## Key decisions

- **Stage 4 timeline:** full schema rewrite happens in Stage 4 with proper transition logic for maintained alarm bits. Phase B does not attempt cross-packet state tracking — those bits log only.
- **Phase B handler shape:** each 0x0200 packet produces one row in `positions`, zero-or-more rows in `events`, plus an UPDATE to `devices.last_seen_at`. All three in a single transaction; rollback on any failure.
- **Event emission policy in Phase B:** one-shot alarm bits (SOS, power off, ignition on/off, vibration, tow, harsh braking, etc.) become event rows. Maintained alarm bits (overspeed, fatigue, GNSS faults, etc.) log only — events come in Stage 4 once per-device state cache exists.
- **Telemetry JSONB keys:** camelCase from `extraMessageParser` output, no transformation. `telemetry.gsmSignal`, `telemetry.batteryPercent`, `telemetry.imei`, `telemetry.wakeUpSource`, etc.
- **`device_id` resolution:** at TCP connection auth time (0x0102), not per-packet. Stored on connection-state object for lifetime of the connection.
- **`last_seen_at` update:** per-packet UPDATE in Phase B. Batched async updates deferred to Stage 4.
- **Scale revisions to Stage 1 schema:**
  - Telemetry cap tightened from 8 KB → **2 KB** (at 100M rows scale, 8 KB average = 800 GB of telemetry storage; 2 KB still comfortably fits realistic Mobicom emissions).
  - GIN index on `positions.telemetry` **required from day 1**, not deferred. Cost to build on empty table is zero; cost to build later on 10M rows is hours of downtime.
  - In-memory LRU cache for device auth — deferred to Stage 4.
  - Batched async `last_seen_at` updates — deferred to Stage 4.
- **Entity file conventions (locked):**
  - EntitySchema (object-based JS), not decorators
  - PascalCase entity `name`, snake_case `tableName`
  - **snake_case property names** for v2 entities (matches schema-v2.md and the newer in-codebase convention)
  - PKs use `bigint` with `generated: "increment"` (matches `bigserial`)
  - No defaults, FKs, indexes, or constraints in entity files — migrations are the single source of truth
  - Tight whitespace (no blank lines between columns)
- **Bigint-as-string in handler code:** all TypeORM bigint columns are treated as JS strings throughout handler code. No `parseInt`. Saved as a feedback memory for the project.
- **Legacy entity registration:** kept in `ormconfig.js` `entities` array through step 4. Strip happens in step 4 alongside handler rewrite, not in step 2 (single-responsibility commits).
- **No side quests until testing phase:** flag adjacent concerns (e.g., `DB_PASSWORD || "12345"` fallback on `ormconfig.js:21`) but don't fix them mid-implementation. Saved as a feedback memory.

## Key findings

- **Real G107 packet captured and decoded** against the in-repo Mobicom V2.2 spec. The audit surfaced Bug 8 (locationExtraParser corruption) and Bug 9 (unhandled real TLVs).
- **`locationExtraParser` uses fictional TLV IDs** (`0x03`, `0x05`, `0x06`) that the Mobicom spec does not define. On real G107 packets these IDs collide with actually-emitted TLVs and **actively corrupt `signal_strength`**. The parser is being deprecated in Phase B; column drops follow in Stage 4.
- **Production database state** (verified earlier this stage): v2 tables (`positions`, `events`, `devices`, `trucks`, `geofences`) present and empty; legacy tables renamed to `*_legacy`; ingestion currently writes **nothing useful** because the legacy entities reference non-existent un-renamed table names.
- **Stage 1 cutover was incomplete in practice.** Stage 1's exit criterion "remove `synchronize: true` from `ingestion/ormconfig.js`" was silently deferred to Stage 2 Phase B. Closed today by `8a1837c`.
- **G107 operates in (mode unknown — needs verification before cutover) connection mode.** Mobicom spec describes both long-connection (continuous TCP + heartbeats every 120s) and short-connection (connect, send, disconnect — battery-preserving). Operational mode affects Phase D load expectations: short-connection at 10k devices reporting every 10 min ≈ 17 auth lookups/sec. Verification task tracked in `phase-b-handler-rewrite.md` §"Pre-cutover verification".
