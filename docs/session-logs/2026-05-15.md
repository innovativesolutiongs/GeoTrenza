## Postscript — Stage 2 closed via simulator-driven verification

Rather than wait on dev, ran `deviceSimulator.js` (the legacy test tool already on EC2) against the live ingestion ourselves. Pre-registered the device (`terminal_id='690106149138'`, `auth_code='4D6F6269636F6D496F54'` = uppercase hex of `"MobicomIoT"`, `model='G107'`).

### Things we hit and fixed forward

1. **`.env` had CRLF line endings.** `bash source` left `\r` on each value (`localhost\r` won't resolve). dotenv inside running pm2 process handled it silently, which is why the server itself ran fine. Fixed with `sed -i 's/\r$//' /var/www/html/gps.geotrenza.com/backend/.env` — idempotent, no runtime impact.

2. **`ipapi.co` rate-limited the EC2 IP.** Simulator's `getLiveLocation()` returned `undefined, undefined`. `Math.round(undefined * 1e6) = NaN`, `.toString(16) = "NaN"`, `padStart` to `"00000NaN"`. Node's `Buffer.from(str, 'hex')` silently truncates at the first invalid char, so server received only 46 of the expected 96-char body. **Parser correctly rejected** with `parseLocationPacket: packet too short (46 hex chars, need >= 82)` and ack result=02 (PARSE_ERROR). **Side benefit: validated the error-categorization wire path on real malformed input.**

3. **Negative longitude (SF: -122.4194) also poisoned the hex.** `Math.round(-122419400).toString(16) = "-74bf8c8"` — minus sign isn't a hex char. Same truncation symptom as #2. Switched to Karachi (24.86°N, 67.00°E — positive both axes, geographically plausible for a Pakistan-operated fleet).

Fix-forward simulator at `/tmp/devsim-fixed.js` on EC2 (ephemeral — re-create if needed). Original `deviceSimulator.js` left untouched.
